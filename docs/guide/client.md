# Client Modes

`FemmClient` is the Python interface for submitting Lua scripts to the py2femm server and receiving results. It supports automatic mode detection and two transport modes: **local** (shared filesystem) and **remote** (REST API).

---

## Basic usage

```python
from py2femm.client import FemmClient

client = FemmClient(mode="remote", url="http://localhost:8082")

result = client.run(lua_script, timeout=120)
print(result.csv_data)      # raw CSV output
print(result.error)          # None if successful
print(result.elapsed_s)      # wall-clock seconds
```

---

## Modes

### Remote (REST API)

The most common mode. The client sends Lua scripts over HTTP to the py2femm server:

```python
client = FemmClient(mode="remote", url="http://192.168.1.100:8082")
```

This works across networks -- the server can be on a different machine.

### Local (shared filesystem)

Used in WSL, where the client and server share the Windows filesystem via `/mnt/c/`:

```python
client = FemmClient(mode="local", workspace="/mnt/c/femm_workspace")
```

The client writes the Lua script to a shared directory, the server (running on Windows) picks it up, and results appear in the same directory.

### Auto-detection

If you omit the `mode` parameter, `FemmClient` auto-detects:

```python
client = FemmClient()  # auto-detect
```

Detection order:

1. `/mnt/c/` exists -- local mode
2. `PYFEMM_AGENT_URL` env var -- remote mode
3. `~/.py2femm/config.yml` with `agent.url` -- remote mode
4. Nothing found -- raises `ConnectionError`

## Configuration

### Constructor arguments

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `mode` | `str | None` | `None` (auto) | `"local"`, `"remote"`, or `None` |
| `url` | `str | None` | `None` | Server URL (remote mode) |
| `workspace` | `Path | None` | `None` | Shared directory (local mode) |

### Environment variable

```bash
export PYFEMM_AGENT_URL=http://myserver:8082
```

### Config file

`~/.py2femm/config.yml`:

```yaml
agent:
  url: http://myserver:8082
```

### `config/default.yml`

The server-side configuration (generated by `setup_femm.bat`):

```yaml
python:
  env_type: conda
  env_name: my_env
  conda_root: C:\ProgramData\Anaconda3
femm:
  exe: C:\femm42\bin\femm.exe
```

## The `run()` method

```python
result = client.run(lua_script: str, timeout: int = 300) -> ClientResult
```

| Parameter | Description |
|-----------|-------------|
| `lua_script` | Complete Lua script as a string |
| `timeout` | Maximum seconds to wait for completion |

Returns a `ClientResult` dataclass:

| Field | Type | Description |
|-------|------|-------------|
| `csv_data` | `str | None` | Raw CSV output from FEMM |
| `error` | `str | None` | Error message if failed |
| `elapsed_s` | `float` | Wall-clock time |

---

## Timeout behavior

If the simulation exceeds the timeout, the server kills the FEMM subprocess and returns an error:

```python
result = client.run(lua_script, timeout=60)
if result.error:
    print(f"Failed: {result.error}")
    # result.csv_data will be None
```

!!! tip
    For parametric sweeps with many configurations, set a generous timeout per simulation (e.g., 120-300s). A single FEMM solve typically takes 2-30 seconds depending on mesh complexity.

---

## Error handling patterns

### Simple

```python
result = client.run(lua, timeout=120)
assert result.error is None, f"FEMM failed: {result.error}"
assert result.csv_data, "FEMM returned no data"
```

### Graceful (for sweeps)

```python
result = client.run(lua, timeout=120)
if result.error or not result.csv_data:
    print(f"Skipping: {result.error}")
    continue
```

## Status check

```python
status = client.status()
print(status)

```

Or from the CLI:

```bash
py2femm status
```
